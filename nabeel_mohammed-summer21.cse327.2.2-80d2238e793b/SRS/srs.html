    1. Introduction

    1.1 Purpose

    This project aims to track the reading progress of students.It will be a virtual classroom where professors and students may communicate with one another and express their questions and concerns through an online platform. Using this application,students can finish their reading task in time. It will explain the system's aim and features, as well as the system's interfaces, what the system will do, the constraints under which it must operate, and how the system will react to various stimulation. This paper is meant for both the system's administrators and developers, and it has been submitted to the appropriate Faculty for approval.


    1.2 Document Conventions

    Prioritize each and every requirement statement.


    1.3 Intended Audience and Reading Suggestions

    This paper is for both administrators and device developers, and it will be submitted to the proper Faculty for approval. This document will also help the owning organization to get basic ideas on the functionality of the project. Chapter 2 of this document contains details of the project functions, users and classes, operating environment, user documentations etc. Chapter 3 contains details regarding the external interface requirements, user interfaces, hardware interfaces. Chapter 4 discusses the system features and Chapter 5 talks about all the non functional requirements.


    1.4 Product Scope

    This software tracks the reading progress of students.This system will be designed to communicate with professors and students in a virtual classroom. It will offer easy and straightforward Instructions for the new users. Additionally, the virtual classroom will be able to generate a unique class code for the system. Students can join a classroom using class code.Teacher can upload reading material in the class with a deadline. Students have to read the material in a given time. Teachers can track their student reading activity with the help of facial recognition.

    The application will be a web based classroom where teachers can track the reading progress of students. Teachers can create classes. Teachers can also review student reading activity in the review reading material section. Student can see reading task and deadline of a particular class but also as a unified class using ‘Reading task and deadline’ section.

    3. External Interface Requirements

    3.1 User Interfaces


    1. Sign In page: Users will be greeted by a sign-in page where he/she can either sign-in. If the user doesn’t log in when he/she tries to open the home page then the sign in page will be shown . Users need to give an email and password to sign in.There are two different buttons in the sign in page: i)Sign in as Teacher ii)Sign in as Student. Teacher have to sign in system through the ‘Sign in as Teacher’ button and student also use the ‘Sign as student’ button to sign in

    2. Sign up page: Teacher or student need to register in the system if they are open the system for the very first time. They have to give some information to sign up, Student have to give their recent photographs. After the sign up system redirects to the sign in page to sign in.

    3. Home page: After the sign in, teacher or student redirect to the home page. If student joined some classes he/she will see the list of classes with ‘go to class’ button. If teacher created sme classes he/she will also see the list of classes with ‘go to class’ button. Student can see reading task and deadline of a particular class but also as a unified class using ‘Reading task and deadline’ button

    4. Create class: Teacher can create class using the ‘create class’ button . Create class button will show when teacher clicks the plus button form the navbar. When teacher clicks create class button, a form will be pop up. Teacher has to give the class room information in the form and click the ‘create’ button. A class will be created with unique class code in the background and link to the teacher id. Then a ‘create class successfully’ notification will show in the home page and teacher can see his/her created class.

    5. Joined class: Student can join a class using the ‘join class’ button . Join class button will show when student clicks the plus button form the navbar. When student clicks create class button, a form will be pop up to enter the class code, student has to give the class code in the form and press the ‘join’ button. Then student can join the class if class code is valid or a ‘no class found’ notification will show in home page. After join a class room, student can see his/her joined class room in the home page.

    6. Class room page: Teacher or student can view the class room page using ‘go to class’ button. In the class room page teacher or student can see the class information. Student can not see the class code but teacher can see the class code and ‘invite student’ button. Teacher can assign material to class using ‘Assign Material to the class’ button but student can not do this.

    7. Send Class code: Teacher can send the class code to the student using ‘Invite student’ button in class room page. Teacher has to give the students emails to send the class code.

    8. Material view: Teacher and student can view the reading material in different way. Student can zoom in and zoom out the material and add different type of question using ‘Add question’ button. Student can also mark a particular part of material.Teacher can view reading material with question list of a page and also see tagged question under the marked text.

    9. Reading review: Teacher can review the reading activity of a student using “Reading review’ button in material page. In reading review page , teacher can see the student images according to the page no. of material. Then, teacher can send his/her review using ‘comment’ button.

    • 3.4 Communications Interfaces

    We are building two applications on two different platforms. But we need work on the same database. For that reason, we need something that handles the job , creating a common way environment. We are using “Rest API” to communicate with both of the platforms. Many applications rely on REST APIs to allow the front end to talk to the back end. If we deploy an Android application, we need an API to allow Android databases to consume information from the database. We might need to convert tabular format into JSON or Serialization. And in this case, we need the API.
